# FORGE — How This Works

This document explains the FORGE system for any Claude Code session or AI agent working in a FORGE-enabled project. Read this before touching anything in `.forge/`.

---

## What FORGE Is

FORGE is a lightweight project state tracker that lives inside each repo. It records commits, tasks, and phase progress into `.forge/forge-data.json`. This data feeds:

- The **FORGE dashboard** — a local HTML UI served at `http://127.0.0.1:7474` on session start
- The **Forge Master** — a nightly AI agent that reads all forge-enabled projects, generates knowledge artifacts, and archives them to `forge-archivist`

FORGE does not replace git. It runs alongside it.

---

## File Structure

```
.forge/
  forge-data.json       ← project state: commits, tasks, phases, activity
  FORGE.md              ← this file
  artifacts/
    *.html              ← visual diagrams (state, architecture) — generated by Forge Master
    decisions/          ← ADRs — generated by Forge Master
    patterns/           ← reusable pattern docs — generated by Forge Master
```

**forge-data.json schema:**
```json
{
  "project": "repo-name",
  "path": "/absolute/path/to/project",
  "branch": "main",
  "updatedAt": "ISO timestamp",
  "filesChanged": 0,
  "commits": [{ "hash": "", "message": "", "timestamp": "" }],
  "activity": [{ "hash": "", "message": "", "timestamp": "", "type": "commit|feat|fix|docs|test" }],
  "phases": [
    { "name": "Setup",       "status": "done|active|pending", "taskCount": 0, "doneCount": 0 },
    { "name": "Core",        "status": "done|active|pending", "taskCount": 0, "doneCount": 0 },
    { "name": "Integration", "status": "done|active|pending", "taskCount": 0, "doneCount": 0 },
    { "name": "Polish",      "status": "done|active|pending", "taskCount": 0, "doneCount": 0 }
  ],
  "tasks": [{ "title": "", "status": "pending|active|done|blocked", "phase": "", "tags": [] }],
  "currentPhase": "Core"
}
```

## How to Update FORGE

### After any git commit — run forge-update

```bash
node ~/.claude/forge/forge-update.js
```

This refreshes `forge-data.json` with the latest commits, file counts, and activity. It is lightweight and safe to run anytime. It only runs if `.forge/forge-data.json` already exists.

The `CLAUDE_PROJECT_DIR` environment variable tells it which project to update. Inside a Claude Code session this is set automatically. Outside a session, run from the project root and it uses `cwd`.

### Full re-initialization

```bash
node ~/.claude/forge/forge-init.js
# or
forge init
```

Use this if forge-data.json is missing, corrupt, or needs a full rebuild from git history.

## The FORGE CLI

`forge` is available globally at `/opt/homebrew/bin/forge`.

```bash
forge status                          # show phase + task summary
forge add "Task title"               # add a task to current phase
forge add "Task" --phase "Core"      # add to specific phase
forge add "Task" --tag feat          # tag: feat | fix | docs | test
forge done "partial title match"     # mark task done
forge active "partial title match"   # mark task active
forge block "partial title match"    # mark task blocked
forge init [path]                    # initialize or rebuild forge-data.json
forge open                           # open FORGE dashboard in browser
```

## The FORGE Dashboard

On Claude Code session start, a local server spins up automatically serving the dashboard at:

```
http://127.0.0.1:7474/forge.html?data=http://127.0.0.1:7474/forge-data.json
```

The server is managed by `~/.claude/forge/forge-serve.js` (global, not in any repo). It stays alive until the browser window is closed. If the port is already in use, it opens the current project in the existing server window.

To open manually: `forge open`

## How the GitHub Sync Works

Each forge-enabled repo contains `.github/workflows/forge-sync.yml`. When you push a commit that touches `.forge/**` on `main`, this workflow:

1. Checks out the source repo
2. Checks out `forge-archivist` using the `ARCHIVIST_TOKEN` secret
3. Copies `.forge/` → `projects/<repo-name>/` in forge-archivist
4. Commits and pushes: `sync(<repo-name>): update .forge from <sha>`

**Requirements for sync to work:**
- The repo must be on GitHub (not just Gitea/local)
- The repo must have an `ARCHIVIST_TOKEN` secret set — a GitHub personal access token with write access to `husky2466-codo/forge-archivist`
- The workflow file must be present at `.github/workflows/forge-sync.yml`

If sync is failing, check GitHub Actions logs first. Common cause: missing or expired `ARCHIVIST_TOKEN` secret.

**Note:** The Forge Master reads forge data locally (from the DevDrive) — it does not depend on GitHub sync. The sync is the off-machine backup path only.

## What the Forge Master Does (and Does Not Do)

The **Forge Master** (`forge-master.js` in `forge-archivist`) runs on a cron schedule:
- Weekdays: 12:00pm
- Weekends: 4:00am

**It WILL:**
- Read your `forge-data.json` and source code (read-only)
- Run `forge-update` to freshen your forge data
- Create and update diagrams in `.forge/artifacts/`
- Push copies of artifacts and forge data to `forge-archivist`
- Update the cross-project knowledge mesh (`mesh.db`)

**It will NEVER:**
- Push commits to your repo
- Edit any file outside `.forge/`
- Delete anything
- Access the backdoor key (that's for Nick + Claude Code sessions only)

Artifacts in `.forge/artifacts/` are generated and managed by Forge Master. You can read them freely. Do not delete them — Forge Master uses them to detect what has already been documented and what is genuinely new.

## Adding FORGE to a New Project

1. Run `forge init` from the project root (or `node ~/.claude/forge/forge-init.js`)
2. Copy `.github/workflows/forge-sync.yml` from `forge-archivist` into the new repo
3. Add `ARCHIVIST_TOKEN` secret to the repo on GitHub (Settings → Secrets → Actions)
4. Copy this `FORGE.md` into `.forge/FORGE.md`
5. Commit everything — the sync will trigger on first push
